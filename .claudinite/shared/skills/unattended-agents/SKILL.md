# Agent architecture principles

Portable rules for *how to structure an unattended (automation-invoked) agent* — the judgment-boundary and write-surface guarantees that keep a workflow-driven agent bounded and reliable, independent of any one project.

- An agent invoked **unattended** by automation (a CI workflow, not interactive Claude Code) should be left only the part of the task that genuinely needs its judgment; hard-code every deterministic part — the setup before *and* the wrap-up after — in the workflow around it. A step described in prose is one the agent can skip, reorder, or improvise; a step the workflow runs in code can't be fumbled, and the shorter prompt keeps the agent on the one thing only it can do. Push that boundary continually: each time a step proves mechanical, move it out of the prompt and into the workflow.
- When a routine needs a capability its own runtime lacks — a credential spanning more repos than the session's allowlist, an execution surface the sandbox can't reach — wrap that part as a **dispatch-only executor** (e.g. a `workflow_dispatch`-only GitHub Actions workflow holding the privileged token) that the routine triggers and awaits. The executor carries **no schedule of its own**: give it a cron and it silently becomes a second orchestrator with a competing trigger. One schedule, owned by the orchestrating routine; executors run only when dispatched. State only the executor's *contract* in the routine's prose (what it computes, what the agent may rely on) — its internal rules live in its code, unlike validators of the agent's *own* output, which the prose must still teach (see the routine-folder rules below).
- **The routine's own in-session steps reach GitHub through the session's MCP tools — never a REST client or a `GITHUB_TOKEN`.** This is the corollary of the rule above: the steps a routine runs *itself* operate over the repos the session was granted, which the MCP GitHub tools already cover, so they need no token. A REST client, a raw `api.github.com` `fetch`, or a `process.env.GITHUB_TOKEN` read in in-session code (a planner, a migration pass, a `run_daily` task, a worker) is the tell that the step was mis-built as if it were the CI *executor* — and it will 401 in the MCP-only session it's scheduled in, silently turning the run into a no-op. Only the part that genuinely needs the broader credential moves to a dispatch-only executor; everything else is MCP. *Diagnosing* whether in-session code can reach a host: probe with a tool that honors the sandbox's `HTTPS_PROXY` — `curl` does — never a bare `node -e "fetch(...)"`, whose global `fetch` ignores `HTTPS_PROXY` and goes **direct**, returning a verdict that bypasses (and so misreports) the proxy's real one.
- Size a routine spec at checklist altitude, and re-compress after every scope addition — specs bloat by accretion, not by any one edit. Keep it to terse imperative bullets stating outcomes, bounds, and judgment criteria; a delegated step gets its contract only, procedure another doc owns gets a pointer (never a restatement), and a why-clause earns its place only where the rule is surprising. Symptoms you've absorbed what belongs elsewhere: the spec re-explains a companion script's logic or a sibling doc's steps, or has grown to several times its sibling routines (the house norm is ~40–90 lines).
- Bound an unattended agent's **write surface**, and enforce it from outside the agent. Pre-create the exact file(s) it may edit, tell it to touch nothing else (no new files, no edits to shared code it might be tempted to "improve"), and after it stops, verify the diff in the workflow — reverting anything outside the allowed set rather than trusting the prompt. Prose is a request; the post-hoc diff check is the guarantee, so a misbehaving agent's blast radius is bounded no matter how it strays. Anything the agent would otherwise reach into belongs inlined in its own file (as a source's local helper), not edited in place.
- In a multi-stage unattended pipeline, converge **every** failure exit — including an infrastructure failure in a pre-agent setup stage (a fetch, an auth step, scaffolding) — to the same human-triage state as an agent-detected failure. A setup-phase failure that reports itself but leaves the original trigger untouched (e.g. a label still set) makes the item look unprocessed rather than blocked, silently defeating the escalation path. Treat every failure mode as equivalent from the triage perspective.
- An unattended agent must **never invoke a tool that surfaces a blocking approval it cannot answer** — there is no human present to click "Allow," so the run stalls indefinitely on step one. The multi-agent-orchestration **Workflow tool is the classic trap**: it mandates an interactive opt-in on *every* run (the prompt only disappears under bypass-permissions / headless modes you cannot rely on the scheduler being in), so a routine that fans out through it hangs before doing any work. Fan out with **plain subagent dispatch** (the Agent/Task tool) instead — it carries no such gate and the orchestrator honors any phase barriers itself — and reserve the Workflow tool for interactive sessions where a human can consent. The general rule: an unattended agent's whole tool path must be answer-free.
- **A change-gated routine must exclude its *own* writes from its triggers, or it feeds itself.** A routine that commits to, files issues in, or emits artifacts into the same repos it monitors will re-fire on its own output unless every trigger filters it out: its own maintenance commits must not count as "the project changed" (match them by bot-author / `[skip ci]` / a subject convention and exclude — and keep that matcher current with the wording the routine actually writes, or a near-miss silently reads as real work); its own standing-tracker/issue updates must not count as activity (a tracker rewritten every run always sits inside a fixed lookback window, so it re-triggers forever); its own generated plan/report artifact must not count as a repo change. The tell is a routine that fires every cycle on a fleet that is actually quiet. Treat "the routine's inputs never include the routine's outputs" as an invariant when wiring any change-detection signal.
- When an unattended routine takes an **irreversible or outward-facing action** (closing an issue, deleting a branch, sending a message), the spec must **define the trigger condition concretely and default to the safe path** — not just name the action. A bare "close if done" invites a wrong close on a superficial match; "close *only* when the ask is verifiably true of the target's current state — else comment" cannot. State what counts as the condition and where to verify it, and make the reversible option (leave it, comment, escalate) the default whenever the check is inconclusive.

## Structuring a routine as prose + scripts

Give a recurring routine its own folder — `dev/routines/<routine-name>/` — holding **`routine.md`** (the standard entry point: the spec the agent reads) beside the deterministic scripts it invokes by path, conventionally `preconditions.sh` and `postconditions.sh` (`bash dev/routines/<name>/preconditions.sh`). This shapes the "hard-code the deterministic parts" boundary above into concrete files, governed by four rules:

- **Bracket the agent's judgment between the two scripts, asymmetrically.** `preconditions.sh` is a cheap eligibility gate — on failure the run **skips**, a clean no-op; size it to the cost of a wasted run (when a no-op is cheap and self-correcting, keep the gate broad and err toward running). `postconditions.sh` validates the output — on failure the run is **terminal: mark it failed, never retry**, because a failed postcondition means the lean spec let the agent produce something invalid; surface it for a human to add guiding prose instead of looping.
- **State in `routine.md` every expectation a script enforces.** A validator (scope diff, metric floor, value check) the prose never taught trips the agent on a bar it was never told; a check should only *confirm* an expectation the agent already held, so map each stated rule to the check that guards it. (Complements the write-surface diff-check above: the diff guarantees the bound, the prose has to teach it.)
- **Extract a script only when it takes a whole deterministic decision off the agent.** Wrapping a still-reasoned step, or a lone command beside an un-scriptable one (a `git push` next to an MCP-only PR-open), is indirection — fold it back into the prose. A helper earns its place by consolidating determinism, not by existing.
- **Keep the job — the feedback loop and judgment criteria — in `routine.md`, never only in a script's comments** (scripts are run, not read). A measure→edit→re-measure loop is job, not a precondition. Rationale for a *deterministic* rule may live in the script it governs; judgment the agent must apply may not. When splitting a routine's prose into files, audit clause-by-clause that no instruction survives only in the deleted original.

# Agentic best practices

Durable, project-agnostic practices for running AI agents — distilled from what worked here. Each is one tight rule; the worked example lives in its own doc.

- **A daily "lessons learned" pass keeps hard-won insights from evaporating.** Run it over recent activity to fold durable, reusable lessons into shared docs before they're forgotten. Key discipline: dedupe ruthlessly against the existing docs, route each lesson to the doc that owns it, keep additions terse — and most days add nothing. Routing into an existing doc is the first choice; when **no** existing doc's scope genuinely fits, a worthy lesson (distilled, portable, durable and reusable, not already covered) earns its own *new* doc — a new category is meant to be learnable, not gated behind extra hurdles. A new doc is for when nothing fits, never a way to dodge folding into an existing owner. (Worked example: `docs/claude/auto-lessons.md`.)
- **Match the agent model to the judgment it must make.** A weaker/cheaper model is adequate for mechanical extraction but fails silently on judgment calls — it ships a plausible-but-wrong output where a capable model would correctly bail. Downgrade only for tasks the weaker model reliably handles. (Downgrading the auto-extractor agent to Haiku led it to ship a bare-title case off a listing page instead of stopping; Sonnet bailed correctly — `docs/claude/auto-extractor.md`.)
- **Give each unattended recurring routine its own standing tracking issue as a self-improvement log, titled exactly `Claudinite tracker: <Name>`.** Find it by that **exact title string, never a fuzzy or partial match** and never a bare number (a number can dangle, and a partial match risks landing on the wrong issue). Log every run that produces a change there — as a **dated comment**, not a sub-issue — so the routine's output accumulates in one reviewable feed, making it easy to audit what it did over time and to tell when the routine itself needs tuning. One issue per routine, not a shared parent. **Create it already closed** if it doesn't exist yet, and **never open, close, or reopen it again afterward** — its state carries no meaning here (it's a log, not a work item), so every run leaves the state exactly as found and only ever appends. When the routine reports a **current state** rather than a stream of discrete changes (a branch/PR report, a status roll-up), keep that state in the issue **body**, rewritten each run so the newest snapshot is at the top, and use the dated comments as the per-run trail — the reader sees today's picture first, not the oldest one. (Worked examples: `docs/claude/auto-lessons.md` #365, `docs/claude/auto-fallback-coverage.md` #366, `docs/claude/auto-branch-report.md` #399.)
- **Keep an unattended routine's instructions in a repo doc, not inlined in the launcher's config.** The launcher prompt (a CC web routine, a cron job's embedded text) should be a thin pointer to a versioned in-repo doc; the doc carries the real spec. Inlined instructions drift silently — they can't be reviewed in a PR, go stale against renamed paths the repo's own tests would have caught, and miss conventions the repo later adds — whereas a doc the repo's checks and lessons pass touches stays current for free.
- **After each task, run a brief efficiency analysis of its tool and process usage — separate from lesson capture.** Ask whether it could have used fewer operations (redundant calls, work that could be batched, polling that could have been a single wait) and less wall time without harming quality (independent serial calls that could have run in parallel, unnecessary sleeps). Flag any process that has already returned its result but is being waited out on shutdown — it can be killed once its output is in hand. Close with one concrete speed-up recommendation, or an explicit "no changes recommended." The bar is high — most runs won't yield one.
- **An orchestration script must validate its injected inputs at entry and fail fast — a silently-missing input degrades every downstream agent invisibly.** A parameterized pipeline (a workflow script reading `args`, a worker handed `targets`, a note carrying config) that proceeds on an undefined input doesn't crash — it interpolates `undefined` into every downstream prompt, and each agent then improvises around the hole, producing plausible output subtly off-target with no error anywhere. Assert the expected shape in the first lines and throw with a message naming the missing field. (A harness gotcha in the same family: pass structured args as actual JSON values, not a JSON-encoded string — a stringified object reaches the script as one string and every field access yields `undefined`; a fan-out panel once reviewed "undefined" in place of the proposal it was convened on, flagged only by the agents themselves.)
- **When a task needs several deferred/lazily-loaded tools (schemas fetched via a tool-search or discovery call rather than present from the start), load them all in one batched query up front, not one per turn.** The full set of tools a flow will need is usually knowable before the first call; fetching each one's schema separately as the need arises costs a round trip per tool for no added information.
- **In an agent config file (e.g. `CLAUDE.md`), `@import` only docs whose guidance applies to essentially every session; point at a specialized or pipeline-specific doc with a plain link instead.** An `@import` is expanded inline at launch in *every* session regardless of relevance, taxing the always-loaded context budget; a plain link is read on demand only when its trigger fires. Reserve `@import` for baseline, always-relevant practice, and route a self-contained pipeline/release/maintenance doc through a soft pointer instead.
- **A SessionStart hook gates work through the session context it injects, not by blocking.** The hook's stdout becomes context the agent reads — the runtime never intercepts it — so enforcement is the agent's instruction-following, not the hook itself. Use it for setup/environment validation: output a STOP directive on failure (the agent asks the user before proceeding); emit nothing on success.
- **For an open-ended visual/layout requirement, render real candidate designs and let the owner pick before building one.** When the spec leaves *how it looks or groups* open, drive the production renderer with hand-shaped fake data and put 2–4 *actually-rendered* options in front of the owner, rather than fully implementing, testing, and snapshot-approving a single interpretation that a later reorientation then discards wholesale. Reach for throwaway mockups through the real render path at the first sign of layout ambiguity — *ahead* of the implement→test→snapshot-approve cycle, which only kicks in once a change already exists.
- **When asked to *show* or *see* a visual artifact, deliver the image into the chat — not a path or a link.** Surface the file itself so it renders inline; a link or bare path makes the owner go fetch it. For a tiny artifact (e.g. a 16/32px icon) also include an exact nearest-neighbor upscale, labelled as enlarged, so the detail stays legible.
- **When a routine regenerates a reviewable artifact (a gallery, a snapshot set, a report), surface it in the chat the same turn you commit it** — a URL to the branch's copy, or the file rendered inline — so review is one click away instead of a hunt. (Complements the show/see rule above: that's for an artifact the owner asked to *see*; this is for one a routine *regenerates*.)
- **When driving a CI-gated merge via API/MCP poll rather than the shell, merge on an already-green required check — don't trigger or wait for a duplicate run.** A push already ran CI on the head commit; opening a PR on the same commit fires an identical second run that adds no signal. If the check isn't yet green, poll on a rolling backoff (e.g. 5 s → 10 s → 15 s → 30 s) until it settles; never tight-poll; and don't subscribe for green — CI success transitions are frequently not delivered as webhook events, so the green signal never arrives.
- **When an action you take kicks off an async downstream process whose output matters to what happens next (a merge triggering a build/publish/deploy), don't report the task done until that process completes and its output exists — poll for it.** Reporting at the trigger point races the async process: a follow-up action taken right after (e.g. "now ship this") would then operate on the stale prior output instead of the one just triggered.
- **In a multi-stage agent pipeline, every failure exit must converge to the same human-triage state.** A setup-phase failure (page download, authentication, scaffolding) that posts a comment but leaves the original trigger label in place makes the item look unprocessed rather than blocked, defeating the escalation mechanism. Treat every failure mode — pre-agent setup, agent bail, post-agent quality check — as equivalent from the triage perspective: all must resolve to the same visible state (e.g. a comment naming what failed, plus the trigger label removed).
